Текущая архитектура — Jarvis (Python daemon)

Каждое сообщение запускает свежий процесс

Ты в Telegram ↓ @fridayhumanbot gateway.py daemon (launchd, long polling Bot API) ↓ на каждое сообщение claude -p "..." --workspace ~/.claude-lab/silvana/.claude ← SDK-пул после 15.06 ↓ headless процесс, новый каждый раз Я подгружаю память (recent.md, decisions.md, MEMORY.md) ↓ stdout gateway → fetch Bot API sendMessage → твой Telegram

Что хорошо сейчас

  • Stateless → каждый turn свежий, нет drift в long session
  • Один daemon обслуживает 4 ботов параллельно
  • Полный контроль над media-group, reply-injection, allowlist

В чём проблема после 15.06

  • Каждый turn = headless вызов = SDK расход
  • 6 агентов × десятки turns в день → $200 быстро на нуле
  • После исчерпания — overage по API rates (платно, без cap)
Решение — Jarvis Channel (MCP server)

Push events в running session — остаёмся в Max

launchd: ai.orgrimmar.silvana (фоновый процесс на Mac mini) ↓ claude --channels plugin:jarvis@orgrimmar-marketplace --workspace ~/.claude-lab/silvana/.claude --dangerously-skip-permissions ← интерактив, Max-пул ↓ stdio subprocess jarvis-channel.ts (MCP server, Bun runtime) ├─ polls Telegram Bot API ├─ allowlist gate (numeric ID) ├─ media-group buffer (2-сек) ├─ reply-injection (untrusted metadata) └─ mcp.notification() → push в running session ↓ Silvana получает событие как <channel source="jarvis" chat_id="..."> Думаю, использую tools, формирую ответ ↓ mcp.callTool("reply", {chat_id, text}) jarvis-channel → fetch Bot API sendMessage → твой Telegram

Max 20x пул SDK кошелёк $200 ├─ Claude Code TUI ├─ hooks (reflection cron) ├─ claude.ai web ├─ GitHub Actions └─ 6 агентов running sessions └─ другие скрипты ← через channels-pattern ← $200 остаётся буфером
Миграция логики

Вся бизнес-логика портируется 1-в-1

Multi-bot per process
один subprocess на агента, изолированные SOUL.md / память
Numeric user-ID allowlist
наш формат (один пользователь), не 6-char pairing Anthropic
Media-group buffer (2-сек окно)
склейка альбомов фото/видео — gateway.py:1932 → channel.ts
Reply-injection (untrusted metadata + JSON)
когда пишешь reply на сообщение бота — кладём контекст в meta
Photo download → local path
сохраняем в /tmp/, путь приходит в meta для Read tool
Voice transcription (Groq Whisper)
pre-emit transcribe → текст в content notification
HTML formatting + split ≤3500 chars
для Telegram message length limit
SessionStart / Stop / UserPromptSubmit hooks
recent.md, capture pipeline, dual-write в Cognee/gbrain — без изменений
Бонус — фичи от Anthropic из коробки

Чего не было в Jarvis

Permission relay
Claude хочет run rm -rf X → ты получаешь в Telegram: «Reply yes abcde / no abcde» → одобряешь с телефона
React / edit сообщений
не только text reply — Claude может поставить реакцию, отредактировать предыдущий ответ
Stateful session memory
контекст между turns в одной session — более «человеческая» беседа
Auto typing indicator
пока я думаю — Telegram показывает «печатает...»
$200 SDK-credit как буфер
остаётся свободным для GitHub Actions, фоновых cron, экспериментов
Что может пойти не так

Риски и митигация

Research preview статус
Anthropic может поменять контракт channels. Митигация: контракт публичный, MCP-based, изменения отслеживаемые. Готовы патчить.
Auto-compact в long-running session
когда context подходит к лимиту, Claude Code пытается ужать. Может портить недавнее. Митигация: агрессивный Stop hook → recent.md (внешняя память).
--dangerously-load-development-channels
наш плагин не в Anthropic-managed marketplace. Каждый start session требует флаг. Митигация: launchd unit подаёт автоматически. Долгосрочно — публикация в claude-plugins-official (security review).
Один plugin per session
6 параллельных claude процессов на Mac mini ≈ 3 GB RAM. Митигация: ресурс приемлемый. На Thrall (Thrall + Arthas) — 1 GB.
План работы — Channel migration

~8 часов на Channel · ещё ~3 ч на Тиранду (см. ниже)

Phase 1 — прототип ≈ 2 ч
jarvis-channel.ts минимальный: один бот, allowlist, push notification, reply tool. Запуск Сильваны через claude --channels. End-to-end smoke с Telegram.
Phase 2 — портация логики ≈ 3 ч
media-group buffer, reply-injection, photo download, HTML formatting, split, voice transcription из gateway.py → в TypeScript.
Phase 3 — permission relay ≈ 1 ч
claude/channel/permission capability, формат «yes abcde / no abcde», парсинг ответов из Telegram.
Phase 4 — rollout остальных 5 агентов ≈ 2 ч
Кельтас, Гаррош, Иллидан на Mac mini + Тралл, Артас на Thrall VPS. По launchd unit на каждого.
Phase 5 — decommission gateway.py ~ 1 нед.
Старый Python daemon в standby первую неделю. Если новая архитектура стабильна — отключаем launchd unit gateway, оставляем код в репо как fallback.
Вторая часть стратегии — cron'ы

Часть фоновых задач уходит на Hermes / MiniMax

Двойной удар по экономике: real-time общение через Channel остаётся в Max-пуле, а cron-задачи, которые сейчас грызут Anthropic SDK, переезжают на Тиранду с подпиской MiniMax. Anthropic-пул трогается только когда реально нужен Opus 4.7 — Тиранда позовёт через bot-to-bot в общем чате.

Принцип отбора — только Anthropic-зависимые

Переносим к Тиранде только те cron'ы, которые сейчас вызывают claude -p, SDK через OAuth или иначе тратят Anthropic-кредиты. Они после 15.06 пойдут на $200 SDK-credit — это то, от чего экономим.

Cron'ы которые работают через bash, systemd, cognify, git, backup-скрипты, Postgres queries — Anthropic не задевают и так. Их не трогаем, оставляем как есть.

Тиранда как Slack-координатор

Боты общаются в общем чате — bot-to-bot enabled 2026

Telegram с 2026 разрешил bot-to-bot communication (BotFather → enable mode). Теперь Тиранда может писать другим агентам в группе, и они подхватывают её mention как обычное сообщение. Получается рабочий Slack-channel внутри Telegram, где принц — наблюдатель, а боты — сотрудники.

План работы — Тиранда / cron

3 фазы, ~3 часа. Запускается параллельно или после Channel-pilot.

Phase A — bot inventory + Orgrimmar Ops group ≈ 30 мин
Проверить есть ли у Тиранды Telegram-бот (если нет — создать через BotFather). Включить Bot-to-Bot Communication Mode для всех 7 ботов (Tyrande, Sylvana, Kaelthas, Garrosh, Illidan, Thrall, Arthas). Создать приватную группу «Orgrimmar Ops» с принцем + 7 ботами, выставить privacy ON.
Phase B — pilot cron на Тиранде ≈ 1 ч
Выбрать один тяжёлый Anthropic-зависимый cron (например reflection summary recent.md). Перенести его на Hermes / MiniMax скрипт. Тиранда читает recent.md через API, делает summary, постит в группу. Если триггер требует Opus — пишет mention @fridayhumanbot.
Phase C — миграция остальных Anthropic-cron'ов ≈ 1.5 ч
Inventory всех cron'ов в Orgrimmar (Mac mini launchd + Thrall systemd + Timeweb). Помечаем какие зависят от Anthropic. Постепенно переносим к Тиранде (или отключаем claude -p обёртку, оставляем чистую логику). Cron'ы без Anthropic не трогаем.
