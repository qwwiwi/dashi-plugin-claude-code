⚠ единственный недоказанный тезис

Channel под launchd = subscription pool?

Anthropic support обещает: «interactive Claude Code in the terminal or IDE» остаётся на subscription, claude -p / Agent SDK / GitHub Actions уходят в SDK $200 credit.

Channels документированы как MCP-серверы, которые пушат события в живую сессию Claude Code. Технически — это interactive. Но публично не подтверждено, как биллинг классифицирует Channel-сессию под launchd без человеческого TTY.

Решение: Phase 0 canary до 15.06 — реальный launchd-запуск, тестовое событие, проверка дашборда + письмо в Anthropic support. Без канарейки рискуем найти проблему 16 июня, когда уже late.

Phase 0 — Critical

Канарейка билинга — без неё не стартуем

phase 0 · canary Доказать, что Channel под launchd = Max pool ~3 ч + 48 ч observation
Цель: получить hard evidence (дашборд + письмо support), что channel-сессия под launchd биллится как interactive subscription, а не как SDK credit.

шаги

1. Минимальный echo-channel
MCP-сервер на bun, capability claude/channel, единственный event "ping" → выводит timestamp в живую сессию.
2. launchd unit на Mac mini
~/Library/LaunchAgents/ai.orgrimmar.canary.plist. Запускает claude --dangerously-load-development-channels plugin:canary@local --channels canary под user session, без TTY.
3. Внешний триггер каждые 5 мин
Cron / отдельный launchd шлёт ping. 48 часов прогона. Логируем токены, latency, ошибки.
4. Проверка дашборда Anthropic
console.anthropic.com → usage → раздельные счётчики subscription vs SDK credit. Канарейка должна крутить subscription, не credit.
5. Письмо в Anthropic support
Описать: launchd, no TTY, --channels плагин, OAuth Max. Прямой вопрос: подпадает ли под "interactive Claude Code"?
6. Gate: PASS / FAIL
PASS = дашборд показывает subscription + support confirm → Phase 1. FAIL = Channel под launchd классифицируется как SDK → пересматриваем стратегию (Tyrande берёт больше).

fallback при FAIL

Если канарейка показала SDK pool: переезжаем на гибрид — Channel запускается из tmux / screen внутри активной user-сессии (true TTY), а не launchd. Тиранда берёт на себя 100% автоматических крон-сценариев. Latency хуже (нужен active login), но Max pool сохраняется.

новое — bot-to-bot safety

Loop guards для Telegram оркестрации

Telegram официально предупреждает: bot-to-bot communication без guards = риск бесконечного цикла (бот A пишет боту B, B отвечает A, A снова отвечает …). При rate spike → блок API на токен.

обязательные guards

Depth-limit: max 3 hops в цепочке
В каждое bot-to-bot сообщение пишем X-Orgrimmar-Depth header. Channel читает, инкрементит, > 3 → drop + alert принцу.
Dedupe по message hash + TTL 60s
Bloom-фильтр / sqlite cache. Тот же текст от того же бота — игнор.
Per-bot rate-limit: 10 msg/min
Token bucket. Свыше — queue с экспоненциальным backoff.
Permission codes only from owner
approve-codes принимаются ТОЛЬКО от user_id 164795011. Никогда от peer бота.
Kill-switch команда
«/halt» от принца в любую группу — все channel-сессии паузятся.
Rollback strategy

Если что-то сломалось

Phase 0 FAIL

Канарейка показала SDK pool. Переключаемся на гибрид: Channel в tmux/screen (true TTY), Тиранда берёт больше крон-логики. Никаких изменений gateway.py.

Phase 1–2 ошибки

Channel прототип не работает. gateway.py не тронут, продолжаем на нём. Откат — просто launchctl unload канала.

Phase 4 cutover regression

Один из агентов потерял сообщения. Per-agent revert: config.json enable агента в gateway → launchctl unload channel-{agent}. Остальные продолжают на Channel.

Phase 5 observation

Через 5 дней замечен дрейф. launchctl load ai.orgrimmar.gateway вернуть в строй. Channel оставить, parallel-mode 1 день, root-cause, retry.
